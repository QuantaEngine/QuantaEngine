# 方案迭代与新方案测试手册

## 1. 优化同一方案

适用于修正公式、替换某层 toy model、增加报告字段，但仍保持 `quantaengine-mvp-v1` 的目标和验收语义。

1. 在原始方案或执行报告中记录要解决的物理/工程问题。
2. 更新 `PLAN_MANIFEST.yaml` 的阈值、场景或产物契约。
3. 先增加能暴露旧行为不足的测试。
4. 只替换目标层，保持相邻 report 接口稳定；若接口必须变化，同时更新所有消费者。
5. 运行目标层测试和全量 pytest。
6. 运行 `execution/run_acceptance.py` 重建证据和参考产物。
7. 更新执行报告、追踪矩阵、API 与物理假设。
8. 检查标准宇宙与失败型宇宙，不得只验证成功案例。

## 2. 测试新宇宙场景

1. 从 `templates/universe_variant.yaml` 复制到 `configs/<scenario>.yaml`。
2. 只覆盖变化参数，并写明科学假设和预期方向。
3. 先运行 `quanta validate-config`。
4. 使用 `quanta compare` 对照标准宇宙。
5. 使用 `quanta scan` 检查结果是否随参数连续、单调或出现可解释阈值。
6. 把新场景加入 manifest 的 `variant_acceptance`。
7. 添加 E2E 测试，禁止只提交手工报告。

## 3. 分叉新方案

当目标、输出契约或科学假设发生根本变化时，不覆盖 v1：

1. 复制目录为新的 `plans/<plan-id>/`。
2. 更换 `plan_id`，状态设为 `draft`。
3. 保留父方案链接并记录差异，而不是复制后失去来源。
4. 给新方案单独定义标准场景、反例、阈值和验收器。
5. 新旧方案必须能在同一代码库中独立复验，直到迁移完成。

## 4. 模型升级优先级

建议按证据收益排序：

1. 为现有 heuristics 增加不确定性和敏感性报告。
2. 用更可靠的轻核/恒星模块替换单一尺度关系。
3. 引入可插拔外部求解器适配层。
4. 最后再扩展化学、生命和文明指标，避免在薄弱物理基础上堆叠复杂评分。

## 5. 完成定义

一次迭代只有同时满足以下条件才算完成：

- manifest 与实现一致；
- 需求追踪无悬空项；
- 单元、集成、E2E、CLI 和扫描测试通过；
- 标准场景和失败场景均有证据；
- 参考报告可由验收器重建；
- 物理假设和不可解释范围已更新；
- 本地提交、两个远端和 CI 指向同一版本。
